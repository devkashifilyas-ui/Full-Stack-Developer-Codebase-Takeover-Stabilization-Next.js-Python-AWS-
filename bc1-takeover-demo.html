Written Audit Summary — Day 1

The BC1 platform was inherited from a terminated vendor with no documentation, no handoff call, and no README. The codebase spans Next.js 14 (App Router), a Python FastAPI backend, .NET Core microservices, and a PostgreSQL data layer deployed to AWS via ECS Fargate.

Initial review reveals hardcoded production credentials in at least two service layers, a CI/CD pipeline with unverified deploy targets, three backend services with undocumented inter-service dependencies, and no structured error handling or alert routing in CloudWatch. Test coverage is below 20% in backend services. The LL97 compliance module is partially wired and should not be treated as production-ready.

Immediate actions required: credential rotation, secrets migration to AWS Secrets Manager, CI/CD deploy path audit, health check fixes on ECS tasks. Once those are resolved, the platform can safely support active feature development alongside the Colombia-based team onboarding.

Section 03
Cold Start Audit Workflow
How a senior engineer enters an unfamiliar codebase with no README and no one to ask.
01

Get the application running locally

Clone the repo and attempt a full local boot. What breaks immediately tells you the state of the build chain, env dependencies, and runtime assumptions. Missing env vars, broken Docker builds, and failed migrations are all signal — not blockers.

signal: env gaps, broken deps, failing migrations
02

Map all services, packages, and external integrations

Walk every directory. Enumerate repos, services, background workers, cron jobs, third-party API clients. Build a dependency inventory. Note what's internal vs external, what talks to AWS, what has no clear owner.

signal: service boundaries, integration points, hidden coupling
03

Inspect CI/CD, Docker, and infrastructure definitions

Review every GitHub Actions workflow, Dockerfile, ECS task definition, and Terraform / CDK config. Understand how the app actually ships to production. Identify dead workflows, hardcoded image tags, missing rollback logic, and IAM gaps.

signal: deploy path, rollback capability, secrets exposure in CI
04

Trace one critical business flow end to end

Pick the most business-critical user flow — project creation, LL97 report generation, EV charging audit — and trace it from UI click through API to database and back. Identify every service it touches, every env var it depends on, and every failure mode.

signal: runtime behavior, undocumented dependencies, fragile paths
05

Produce a risk-ranked written audit

Document everything found — immediate risks (security, broken deploys), short-term fixes (test coverage, missing alerts), and deferred cleanup (refactors, technical debt). This becomes the single source of truth for the remediation plan and handoff to the Colombia-based team.

deliverable: written audit report + prioritized risk register
Section 04
Risk Register
Risks identified during initial cold audit, ranked by severity and business impact.
Hardcoded Production Credentials
Critical
Database passwords, API keys, and AWS credentials found directly in source code and CI workflow files. Any access to the repo = full production access.
Fix: Rotate immediately → migrate to Secrets Manager + SSM
ECS Health Checks Failing
Critical
Two ECS Fargate task definitions have misconfigured health check paths. Tasks are cycling or silently running in degraded state without alerting.
Fix: Correct health check endpoints + verify task role permissions
Broken Database Migrations
Critical
Three Alembic migrations are out of sequence. Schema state in staging diverges from production. Running a clean migration chain will fail without manual intervention.
Fix: Reconcile migration chain + snapshot prod schema
No Rollback Logic in CI/CD
Critical
GitHub Actions deploy workflow pushes directly to ECS with no automated rollback on health check failure. A bad deploy will silently degrade production.
Fix: Add health-gate + auto-rollback step to deploy workflow
Stale Dependencies
Medium
14 npm packages and 6 Python packages are 2+ major versions behind with known CVEs.
Fix: Audit + incremental upgrade plan
No CloudWatch Alarms
Medium
Logs are flowing to CloudWatch but zero alarms are configured. No alerting on errors, latency, or task failures.
Fix: Define alarm baselines + SNS routing
Undocumented .NET Services
Medium
Three .NET Core microservices have no OpenAPI specs, no Swagger docs, and unclear ownership within the system.
Fix: Trace + document service contracts
Test Coverage < 20%
Low
Backend service test coverage is minimal. Refactors and new features carry silent regression risk.
Fix: Coverage baseline + test plan for critical paths
No Onboarding Documentation
Low
No README, no architecture docs, no env setup guide. Colombia team cannot onboard without a written guide.
Fix: Create in Week 3 documentation sprint
Mobile API Contracts Missing
Low
Planned Android/iOS app has no defined API contracts. Mobile developer cannot start without OpenAPI specs.
Fix: OpenAPI export after service audit complete
Section 05
Secrets & Production Safety
Exact remediation flow when hardcoded credentials are found in an inherited codebase.
🔍
Detect
Scan source, git history, CI logs for hardcoded values
⚡
Assess
Verify credentials are active. Determine blast radius
🔄
Rotate
Immediately invalidate and regenerate all exposed credentials
🔒
Migrate
Move all secrets to Secrets Manager or SSM Parameter Store
✅
Validate
Verify services load secrets correctly in all environments
# secrets_loader.py — safe secret resolution pattern import boto3 import json def get_secret(secret_name: str, region: str = "us-east-1") -> dict: client = boto3.client("secretsmanager", region_name=region) response = client.get_secret_value(SecretId=secret_name) return json.loads(response["SecretString"]) # Usage — never hardcode credentials secrets = get_secret("bc1/prod/database") DATABASE_URL = f"postgresql://{secrets['user']}:{secrets['password']}@{secrets['host']}/bc1"
# ssm_config.py — environment config via SSM Parameter Store import boto3 def get_param(name: str, decrypt: bool = True) -> str: ssm = boto3.client("ssm") response = ssm.get_parameter(Name=name, WithDecryption=decrypt) return response["Parameter"]["Value"] SENTRY_DSN = get_param("/bc1/prod/sentry_dsn") CLICKUP_TOKEN = get_param("/bc1/prod/clickup_token") AWS_S3_BUCKET = get_param("/bc1/prod/s3_bucket")

Incident Response Playbook
01

Detect

CloudWatch alarm triggers → SNS → PagerDuty / Slack. Sentry captures exception with full stack trace.

02

Assess

Check ECS task health, CloudWatch error rate, recent deploys. Is this a code regression or infra issue?

03

Contain

If deploy-related: roll back ECS service to previous task definition. If data-related: isolate the flow.

04

Resolve & Document

Fix root cause, deploy fix, verify health gates pass, write incident report with timeline and remediation.
